## 🧠 О проекте
**Project Genesis** — это production-ready инфраструктурное AI-ядро, не зависящее от предметной области (domain-agnostic). Это не учебный скрипт, а масштабируемый инженерный MLOps-актив.

Система реализует паттерн **детерминированной мультиагентной оркестрации** («контролируемая эмерджентность»).
* **Интеллект** распределен между изолированными, специализированными LLM-агентами (stateless).
* **Власть** (управление потоком, лимиты, память и циклы) строго контролируется детерминированным кодом (роутером) на Python.

Это инженерный подход к ИИ: никаких галлюцинирующих бесконечных циклов в стиле AutoGPT. Только предсказуемые, контролируемые и надежные AI-пайплайны.

## 🏗 7 Архитектурных слоев

Ядро строго разделено на 7 абстрактных слоев. Инструменты могут меняться, абстракции — никогда.

1. **Execution / Runtime Layer (Плоскость исполнения)**
   * *Где система живет.* Обеспечивает изоляцию, воспроизводимость и переносимость (Docker, Docker Compose).
2. **Data & State Layer (Память и состояние)**
   * *Что система помнит.* Разделено на факты (PostgreSQL), семантическую базу знаний (Qdrant) и оперативный контекст (JSON/Redis).
3. **Ingestion & Transformation Layer (Пайплайн данных)**
   * *Как внешний мир попадает внутрь.* ETL-процессы, очистка, чанкинг (нарезка) и векторизация. Здесь закладывается 80% качества системы.
4. **Intelligence Units (Агенты)**
   * *Кто думает.* Независимые исполнители (Аналитик, Критик, Ресерчер). Они не имеют глобального контекста и выполняют строго одну функцию.
5. **Orchestration & Control Layer (Оркестратор)**
   * *Кто управляет.* Детерминированный мозг системы (не ML). Маршрутизирует задачи, контролирует порядок вызова агентов и лимиты.
6. **Feedback & Evaluation Layer (Иммунная система)**
   * *Кто проверяет.* Оценка результатов работы агентов. Включает агента-Критика, логику reject/retry (отклонить/повторить) для защиты от галлюцинаций.
7. **Interface & Integration Layer (Адаптеры)**
   * *Контакт с человеком.* API (FastAPI), боты (Telegram). Интерфейс не содержит бизнес-логики, это просто "кожа" системы.

## 🚧 Границы ядра (Что НЕ входит в систему)

Чтобы ядро оставалось универсальным, следующие вещи жестко исключены из базовой архитектуры:
* ❌ **Бизнес-логика компаний:** Ядро не знает, что такое «VR-клуб» или «E-commerce». Бизнес-правила натягиваются поверх ядра через конфигурационные файлы (скины).
* ❌ **UI / UX и дизайн:** Ядро — это бэкенд-движок.
* ❌ **AGI / Полная автономия:** Агенты не могут сами ставить себе глобальные цели. Всё ограничено Оркестратором.
* ❌ **Обучение LLM с нуля:** Мы управляем интеллектом (Inference), а не выращиваем базовые модели (Pretraining).

## 🗺 План строительства (10 недель)

Проект реализуется по строгому инженерному графику:

* **Неделя 1:** Фундамент исполнения (Docker, БД, структура репозитория)
* **Неделя 2:** Данные и состояние (Схемы БД, трекинг состояния агентов)
* **Неделя 3:** Ingestion Pipeline (Парсинг, чанкинг, эмбеддинги)
* **Неделя 4:** Базовый Агент (Классы, жесткие контракты ввода/вывода)
* **Неделя 5:** Мультиагентное взаимодействие (Связка Аналитик -> Критик)
* **Неделя 6:** Ядро Оркестрации (FSM, Роутер, Лимиты)
* **Неделя 7:** Иммунная система (Оценка, логика повторов)
* **Неделя 8:** Слой интерфейсов (FastAPI / Telegram Bot)
* **Неделя 9:** Упаковка сценариев (Натяжка первого бизнес-скина)
* **Неделя 10:** Стабилизация и документация (Релиз MVP)

## 📂 Структура репозитория

```text
project-genesis/
├── docker-compose.yml          # Уровень 1: Оркестрация сервисов
├── Dockerfile                  # Уровень 1: Сборка контейнера
├── .env.example                # Шаблон переменных окружения
├── requirements.txt            # Зависимости Python
├── README.md                   # Документация (EN)
├── README_RU.md                # Документация (RU)
├── configs/                    # Уровень конфигурации (Скины)
│   ├── core_config.yaml        # Базовые лимиты, параметры моделей
│   ├── agents_config.yaml      # Роли агентов, промпты
│   └── scenarios/              
│       └── vr_club_skin.yaml   # Демо-скин: Правила для VR-клуба
├── src/                        # Исходный код
│   ├── main.py                 # Точка входа (FastAPI)
│   ├── interface/              # Уровень 7: Интерфейсы и интеграция
│   ├── orchestration/          # Уровень 5: Оркестрация и контроль
│   ├── agents/                 # Уровень 4: Интеллектуальные единицы
│   ├── evaluation/             # Уровень 6: Оценка и обратная связь
│   ├── ingestion/              # Уровень 3: Сбор и трансформация данных
│   ├── memory/                 # Уровень 2: Данные (Семантика)
│   └── storage/                # Уровень 2: Данные (Персистентность)
└── tests/                      # Тесты
```

## 🚀 Быстрый старт (Для разработчиков)
1. Скопируйте `.env.example` в `.env` и укажите ваши API ключи.
2. Запустите `docker-compose up --build -d`.
3. Документация API будет доступна по адресу `http://localhost:8000/docs`.
