# THE ARCHITECT'S CODEX

## ГЛАВА 1: ИНФРАСТРУКТУРА И СРЕДА (EXECUTION LAYER)

### 🧠 Архитектурные принципы (The Why)
*   **Контейнер ≠ Виртуалка:** Контейнер не имеет своего ядра ОС. Он использует ядро хоста (Linux/WSL). Поэтому он весит мегабайты и стартует за секунду.
*   **Stateless (Эфемерность):** Контейнер — это расходник. Его можно убить в любой момент. Все ценные данные (State) должны лежать строго во внешних `volumes`.
*   **Infrastructure as Code (IaC):** Мы используем `docker-compose.yml`, чтобы архитектура была декларативной (описана кодом), версионировалась в Git и поднималась одной кнопкой на любом сервере.
*   **Правило версий:** Никогда не использовать тег `:latest` в проде. Версии фиксируются жестко (например, `postgres:15-alpine`), чтобы избежать внезапных поломок при обновлениях.

### 🛠 Базовый Стек и Инструменты (The How)
*   **WSL 2:** Нативная среда Linux внутри Windows. Код должен лежать в файловой системе Linux (`~/projects/...`), иначе будут тормоза и баги с правами.
*   **PostgreSQL (Слой State):** Реляционная БД. Хранит точные факты, логи, юзеров. Нагрузка: I/O Bound.
*   **Qdrant (Слой Memory):** Векторная БД. Хранит эмбеддинги (числовые векторы) для поиска по смыслу (RAG).

### 💻 Боевые команды (Muscle Memory)
*   `docker compose up -d` — поднять всю инфраструктуру в фоновом режиме.
*   `docker compose down` — убить контейнеры и сеть (данные в volumes выживут).
*   `docker compose ps` — проверить статус запущенных сервисов.
*   `docker compose logs -f <service_name>` — читать логи сервиса в реальном времени.
*   `source venv/bin/activate` — войти в изолированную комнату Python-библиотек.

### 🛡 Гигиена Разработчика
1.  **Секреты:** Пароли и ключи живут ТОЛЬКО в файле `.env`. Файл `.env` ОБЯЗАТЕЛЬНО добавлен в `.gitignore`.
2.  **Git SSH:** Авторизация на GitHub идет через SSH-ключи (`ed25519`), а не через пароли по HTTPS.
3.  **Коммиты:** Используем префиксы (`feat:`, `fix:`, `chore:`) для читаемости истории.

## ГЛАВА 2: PYTHON И ЗАВИСИМОСТИ (APPLICATION LAYER)

### 🧠 Архитектурные принципы
*   **Изоляция зависимостей:** Никогда не ставим пакеты глобально. Всегда используем `venv`.
*   **Детерминированность сборок:** В `requirements.txt` версии фиксируются жестко (`==`). Команда `pip freeze > requirements.txt` — лучший друг. Обновление библиотек (особенно AI-стека) делается только осознанно и с тестированием.
*   **Чистота архитектуры:** Утилиты, тесты инфраструктуры и миграции лежат в папке `scripts/`. В папке `src/` лежит только ядро системы.
*   **Strict Directory Structure (Строгая структура):** Никогда не плодить ad-hoc папки (например, `scripts/`), если в утвержденной архитектуре уже есть каноничная директория для этого домена (например, `tests/`). Инструментарий (Cursor) видит весь контекст проекта, задача архитектора — поддерживать чистоту этого контекста.

### 💻 Боевые команды
*   `python -m pip install --upgrade pip` — обновление установщика (делать при создании venv).
*   `pip install -r requirements.txt` — установка всех зависимостей проекта.

## ГЛАВА 3: PRAGMATIC OOP (ПРАГМАТИЧНОЕ ООП)

Мы не строим сложные иерархии наследования. Мы используем ООП исключительно для **инкапсуляции состояния** (чтобы не таскать переменные вроде подключения к БД по всему коду).

### Словарь Архитектора:
1. **Класс (Class):** Чертеж. Инструкция по сборке. (Например, `class PostgresManager:`).
2. **Объект (Instance):** Физическая сущность в памяти, созданная по чертежу. (`db = PostgresManager()`).
3. **Метод (Method):** Функция, живущая внутри класса. Кнопка действия. (`db.execute_query()`).
4. **Атрибут (Attribute):** Переменная, живущая внутри объекта. Его внутренняя память. (`self.conn`).
5. **`__init__` (Конструктор):** Скрипт первого запуска. Срабатывает автоматически при создании объекта.
6. **`self`:** Указатель объекта на самого себя. Способ сказать "положи это в МОЮ память, а не в общую". Питон подставляет его в методы автоматически.

## ГЛАВА 4: SEMANTIC MEMORY (ВЕКТОРНЫЕ БАЗЫ)

### 🌌 Анатомия Векторного Пространства
*   **Collection (Коллекция):** Пространство с фиксированным количеством измерений (например, 1536). Аналог таблицы в SQL.
*   **Point (Точка):** Единица данных. Состоит из: `ID` + `Vector` (координаты) + `Payload` (метаданные: текст, дата, JSON).
*   **Distance Metric (Метрика):** Способ вычисления "похожести" смыслов. Мы используем `Cosine` (Косинусное сходство) — измеряет угол между векторами.
*   **HNSW:** Алгоритм под капотом Qdrant, который строит графы для сверхбыстрого поиска по миллионам векторов без полного перебора.





## 🛠 БОЕВАЯ ШПАРГАЛКА (MUSCLE MEMORY)
*Этот раздел — выжимка консольных команд для ежедневной работы. Пополняется каждую неделю.*

### 🐍 Python & PIP (Управление пакетами)
*   `source venv/bin/activate` — войти в виртуальное окружение (или алиас `genesis`).
*   `python -m pip install --upgrade pip` — обновить сам установщик (делать сразу после создания venv).
*   `pip list` — посмотреть все установленные библиотеки и их версии в текущем окружении.
*   `pip freeze > requirements.txt` — **магическая команда**: собрать все текущие пакеты с их точными версиями и записать в файл.
*   `pip install -r requirements.txt` — установить все зависимости из файла разом.

### 🐳 Docker (Управление инфраструктурой)
*   `docker compose up -d` — поднять всё в фоне.
*   `docker compose down` — убить контейнеры и сеть (данные в volumes останутся).
*   `docker compose ps` — посмотреть статус запущенных контейнеров.
*   `docker logs -f <имя_контейнера>` — прилипнуть к логам контейнера в реальном времени (выход: `Ctrl+C`).
*   `docker exec -it <имя_контейнера> bash` (или `sh`) — зайти "ногами" внутрь работающего контейнера.

### 🐘 PostgreSQL (Утилита psql)
*   `docker exec -it project-genesis-postgres-1 psql -U genesis -d genesis_db` — зайти прямо в консоль базы данных.
*   `\dt` — (внутри psql) показать список всех таблиц.
*   `\d <имя_таблицы>` — (внутри psql) показать структуру конкретной таблицы (колонки, типы данных).
*   `\q` — (внутри psql) выйти из базы обратно в терминал Linux.

### 🐙 Git (Повседневный флоу)
*   `git status` — проверить, какие файлы изменены.
*   `git add .` — подготовить все изменения к коммиту.
*   `git commit -m "тип: описание"` — зафиксировать изменения (типы: `feat`, `fix`, `chore`, `refactor`).
*   `git push` — отправить код на GitHub.
