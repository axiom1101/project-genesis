# ТЕХНИЧЕСКОЕ ЗАДАНИЕ (SRS)
**Проект:** Универсальное AI-ядро с детерминированной мультиагентной оркестрацией (Project Genesis)  
**Роль исполнителя:** AI Infrastructure Engineer / MLOps Engineer  
**Статус документа:** Утверждено (Базовый план на 10 недель)  

---

## 1. Общие сведения о системе
### 1.1. Назначение системы
Разработка production-ready инфраструктурного AI-ядра (AI Pipeline + Agent Swarm), предназначенного для автоматизации бизнес-процессов, аналитики данных и принятия решений. Система строится по принципу «контролируемой эмерджентности»: интеллектуальная работа выполняется изолированными LLM-агентами, а управление потоком выполнения (routing) осуществляется строго детерминированным кодом.

### 1.2. Бизнес-контекст и масштабируемость
Ядро системы является **domain-agnostic** (не зависит от предметной области). 
В качестве MVP (Proof-of-Value) реализуется вертикаль «Управление компьютерным клубом / VR-ареной» (анализ загрузки, выявление аномалий в работе администраторов, прогнозирование оттока). Бизнес-логика реализуется исключительно на уровне конфигурационных файлов (скинов) и не зашивается в базовый код ядра.

---

## 2. Архитектурный паттерн и технологический стек
Система реализует гибридный паттерн: **Workflow-Oriented Core + Agent-Oriented Processing**.

**Маппинг абстракций на реализации (Стек MVP):**
*   **Compute (Среда исполнения):** Python 3.x (async).
*   **Execution / Runtime (Изоляция):** Docker, Docker Compose.
*   **State Storage (Оперативное состояние и факты):** PostgreSQL.
*   **Semantic Memory (Долгосрочная память):** Vector DB (Qdrant / FAISS).
*   **Orchestration (Управление потоком):** Кастомный Python-роутер (FSM - конечный автомат) / LangGraph.
*   **Boundaries (Интерфейсы):** REST API (FastAPI), Telegram Bot API.

## 3. Логическая архитектура ядра (7 уровней)

### Уровень 1: Execution / Runtime Layer (Плоскость исполнения)
*   **Функция:** Обеспечение воспроизводимости, изоляции и переносимости системы.
*   **Требование:** Архитектура не должна меняться при смене среды запуска (локально -> сервер -> облако).

### Уровень 2: Data & State Layer (Слой данных и состояний)
*   **Функция:** Хранение контекста. Разделяется на 3 контура:
    1.  *Persistent State:* Факты и логи (PostgreSQL).
    2.  *Semantic Memory:* Векторная база знаний (RAG-паттерн).
    3.  *Operational State:* Краткосрочный контекст текущей задачи (JSON/Redis).

### Уровень 3: Ingestion & Transformation Layer (Пайплайн данных)
*   **Функция:** Прием данных из внешнего мира (файлы, API), очистка, нормализация, чанкинг (нарезка) и векторизация (embedding).

### Уровень 4: Intelligence Units (Агентный слой)
*   **Функция:** Исполнение узкоспециализированных задач.
*   **Требование:** Агенты (Analyst, Critic, Researcher, Executor) не имеют глобального контекста. Каждый агент — это изолированная функция (stateless), принимающая input и отдающая output согласно промпту. Отсутствует единое «Я» системы.

### Уровень 5: Orchestration & Control Layer (Слой управления)
*   **Функция:** Детерминированная маршрутизация задач.
*   **Требование:** Жесткий код (не ML). Определяет порядок вызова агентов, лимиты, циклы и стоп-критерии. Интеллект находится в агентах, власть — в оркестраторе.

### Уровень 6: Feedback & Evaluation Layer (Иммунная система)
*   **Функция:** Оценка качества выдачи (Self-reflection / Self-improvement).
*   **Требование:** Наличие агента-Критика (Critic Agent) и правил валидации. Система должна уметь отклонять (reject) галлюцинации и инициировать повторные попытки (retry) до выдачи результата пользователю.

### Уровень 7: Interface & Integration Layer (Слой адаптеров)
*   **Функция:** Точка входа/выхода.
*   **Требование:** Интерфейс (UI/Bot/API) не должен содержать бизнес-логики.

## 4. Функциональные требования (FR)
1.  **FR-1 (Ingestion):** Система должна принимать сырые данные (Excel, PDF, текст), очищать их и сохранять в векторную БД.
2.  **FR-2 (Orchestration):** Система должна маршрутизировать задачу между агентами на основе конфигурационного файла (YAML/JSON).
3.  **FR-3 (Evaluation):** Система должна автоматически проверять результат работы агента-исполнителя и, при несоответствии метрикам качества, отправлять задачу на доработку.
4.  **FR-4 (Memory):** Система должна извлекать релевантный контекст из долгосрочной памяти (Vector DB) перед передачей задачи агенту.
5.  **FR-5 (Configuration):** Смена ролей агентов, промптов и сценариев поведения должна осуществляться через изменение конфигурационных файлов без перекомпиляции/переписывания кода ядра.

---

## 5. Нефункциональные требования (NFR)
1.  **NFR-1 (Детерминированность):** Исключены бесконечные циклы (infinite loops). Оркестратор должен иметь жесткие лимиты на количество итераций (max_retries).
2.  **NFR-2 (Наблюдаемость / Observability):** Все переходы состояний (State transitions) и решения агентов должны логироваться для обеспечения возможности дебага.
3.  **NFR-3 (Безопасность):** Агенты не имеют прямого неконтролируемого доступа к ресурсам (выполнение кода, деплой) без прохождения слоя Evaluation (ревью).

---

## 6. Границы системы (Out of Scope)
Следующие элементы **исключаются** из текущего скоупа разработки ядра:
1.  **Бизнес-логика в коде:** Ядро не знает о существовании «клубов» или «VR-арен». Это реализуется через конфигурацию (скин).
2.  **UI / UX:** Разработка фронтенда, дашбордов и сложного дизайна не производится. Интерфейс — утилитарный (API / CLI / Telegram).
3.  **Highload / Kubernetes:** На этапе MVP автомасштабирование и SRE-практики не внедряются.
4.  **AGI / Полная автономия:** Свободная постановка целей агентами (в стиле AutoGPT) запрещена.
5.  **Обучение моделей (Pretraining):** Система использует готовые LLM (через API или локальный инференс). Обучение моделей с нуля не производится.

## 8. Вектор дальнейшего масштабирования (Post-MVP)
После фиксации ядра (Неделя 10), масштабирование производится исключительно в виде **надстроек**, не нарушающих целостность ядра:
*   *Inference Extensions:* Подключение локальных ML-моделей.
*   *Action Agents:* Добавление агентов с правом выполнения действий (Tool-using agents) под строгим контролем супервизора.
*   *External Integrations:* Подключение платежных шлюзов, CRM.
*   *DevOps:* Перенос в Kubernetes, настройка CI/CD пайплайнов.
